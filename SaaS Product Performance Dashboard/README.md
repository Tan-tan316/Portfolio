## Про проєкт

Стратегічний дашборд для продуктового менеджера мобільної гри, який відповідає на три ключові питання:

- Чи варто продовжувати інвестувати в продукт?
- Де ми втрачаємо гроші?
- Хто наш ідеальний платний користувач?

**Період даних:** березень – грудень 2022 (10 місяців)  
**Стек:** PostgreSQL (DBeaver) · Tableau Public  
**Термін виконання:** 7 днів

---

## Джерела даних

| Таблиця | Опис | Рядків |
|---------|------|--------|
| `games_payments` | Транзакції: user_id, game_name, payment_date, revenue_amount_usd | 272K |
| `games_paid_users` | Атрибути: user_id, game_name, language, age, has_older_device_model | 56K |

**Модель даних:** разові мікротранзакції (5+ платежів на користувача на день). Юзер вважається "платним" у місяці якщо зробив хоча б один платіж.

---

## SQL — архітектура запиту

Весь аналіз побудований на одному запиті з 10 CTE.

```
games_payments + games_paid_users
        ↓
1. GENERATE_SERIES → місячна сітка (березень–грудень 2022)
        ↓
2. CROSS JOIN → всі комбінації user × game × місяць
        ↓
3. LEFT JOIN → фактичні платежі (NULL = не платив у цьому місяці)
        ↓
4. LAG + накопичувальна сума → revenue минулого місяця + сума за весь час
        ↓
5. CASE WHEN → статус користувача
        ↓
6. LEFT JOIN games_paid_users → мова, вік
        ↓
7. Фінальні метрики: LTV, Lifetime, ARPPU
```

### Логіка класифікації статусів

| Статус | Умова |
|--------|-------|
| **New** | revenue > 0, минулого місяця = 0, сума за весь час = 0 |
| **Retained** | revenue > 0, минулого місяця > 0 |
| **Reactivated** | revenue > 0, минулого місяця = 0, але сума за весь час > 0 |
| **Churned** | revenue = 0, минулого місяця > 0 |

Ключове рішення — використання `GENERATE_SERIES` для побудови повної місячної сітки. Це дозволяє коректно відстежувати Reactivated користувачів і Churn через порожні місяці, чого не можна зробити через `LAG/LEAD` на вихідних даних.

## Ключові висновки

**Ріст є, але сповільнюється**  
MRR зростав з $1.5K у березні до піку $9.3K у жовтні, після чого просів до $8.4K у грудні. Кількість нових користувачів стабільна (~35 на місяць), що свідчить про стабільну роботу маркетингу.

**Retention — сильна сторона**  
Березнева когорта через 9 місяців утримала ~42% користувачів. База Retained стабільно зростає щомісяця. В останньому кварталі збільшилась кількість Reactivated — гра навчилась повертати тих, хто пішов.

**Тривожний сигнал: Contraction > Expansion**  
На початку Expansion MRR стабільно перевищував Contraction — гравці охоче платили більше. В останньому кварталі ситуація змінилась: Contraction перевищив Expansion. При зростаючій базі MRR впав — проблема в монетизації, не в залученні.

**Цільова аудиторія: 14–25 років**  
Найвищий середній LTV у наймолодших категоріях ($284 і $263). Маркетинг і контент варто фокусувати на цій аудиторії.

## Складнощі та уроки

**Неправильне джерело даних**  
Два дні цифри були в 3.6× більші за еталонні. Причина: SQL звертався до таблиці тільки за назвою і брав дані з іншої таблиці в базі. Вирішилось вказавши повний шлях `schema.table_name`.  
→ *Урок: завжди вказуй повний шлях до таблиці*

**KPI-картки з порівнянням MoM**  
LOOKUP або сумував всі місяці до обраного, або показував тільки останній при виборі діапазону дат. Після кількох днів роботи дійшла висновку: порівняння MoM — функція операційного дашборду, а не стратегічного.  
→ *Урок: спочатку визнач тип дашборду, потім вибирай метрики*
